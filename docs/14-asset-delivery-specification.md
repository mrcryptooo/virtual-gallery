# 14 — Asset Delivery Specification

**Status: LIVING (2026-07-09).** The definitive production guide for creating and delivering assets to Virtual Gallery. Artist-facing: no code knowledge assumed. It restates the frozen contracts (docs 07/08 and the manifest schema) in production terms — where this document and a frozen document could ever disagree, the frozen document wins and this one gets corrected.

**The one-sentence version:** deliver lossless 2:1 sRGB equirectangular PNGs named with permanent kebab-case slugs into `content-src/<project>/panos/`, with consistent camera height and locked exposure per project — everything else is generated by the pipeline.

---

## 1. Panorama Masters

The master renders are the product. Everything the visitor ever sees is generated from these files — deliver them at the highest quality Lumion can produce.

| Property | Requirement |
|---|---|
| **Required resolution** | **16384 × 8192** (16K) — the target for all hero content |
| **Acceptable resolutions** | 8192 × 4096 (8K) minimum — the validator **rejects anything smaller**. Between 8K and 16K, non-standard widths (e.g. 12000) are accepted but waste pixels: tile resolution rounds *down* to the nearest power-of-two ladder (12000-wide → same tiles as 8K). Stick to exactly 8K or 16K. |
| **Projection** | Equirectangular 360°, exactly **2:1 aspect ratio** (hard-rejected otherwise). Full sphere — no partial/cylindrical panoramas. |
| **Bit depth** | 8-bit per channel (24-bit RGB). No 16-bit, no HDR/EXR in v1. |
| **Color space** | **sRGB.** No Adobe RGB, no ProPhoto, no wide-gamut profiles — the validator rejects non-sRGB color spaces. |
| **File format** | **PNG (preferred — lossless).** Highest-quality JPEG (quality 95+) accepted; TIFF accepted. Never deliver an already-compressed low-quality JPEG — the pipeline re-encodes, and compression on top of compression shows. |
| **Compression** | PNG: any (lossless by definition). JPEG: minimum quality 95, 4:4:4 chroma if the exporter offers it. |
| **Naming convention** | Filename = the panorama's **permanent id**: kebab-case, `a–z 0–9 -` only — `entry-hall.png`, `living-room-day.png`. See §3 — these become URLs and never change. |
| **Camera height** | **1.60 m** eye level, identical in every scene of a project (±5 cm). Inconsistent heights break the step-through illusion between scenes. Exceptions (balcony overlook, stair mid-landing) are deliberate choices, not accidents. |
| **Camera rotation** | Camera perfectly **level** — zero roll, zero pitch on the export camera. The horizon must sit exactly on the vertical center line of the equirect. Yaw (which way "forward" faces) is free; you'll author each scene's initial view later, but a consistent forward direction per project makes hotspot authoring easier. |
| **Exposure consistency** | **Locked manual exposure per project.** Disable auto-exposure / eye adaptation. Scene-to-scene exposure jumps are plainly visible during transitions and read as an error. Interior/exterior pairs may use two fixed exposure setups — but fixed, not auto. |
| **White balance** | **Locked per project.** One WB setting for all scenes sharing lighting; never auto. |

### Lumion rendering recommendations

- Maximum quality preset (highest star rating / ray-tracing quality your version offers) — these renders are re-encoded and zoomed into; source quality is unrecoverable later.
- Render at final resolution directly. Never upscale a smaller render to 16K.
- Full lighting/GI enabled; bake time is worth it — this is a portfolio.
- Disable every screen-space camera effect that breaks in 360 (see mistakes list below).

### Quality checklist (per panorama, before delivery)

- [ ] Exactly 2:1, at 16384×8192 or 8192×4096
- [ ] Horizon level (open the image: the horizon is a straight line at exactly half height)
- [ ] No visible seam at the left/right image edge (yaw ±180 joins seamlessly)
- [ ] Zenith and nadir clean — look straight up and straight down in any 360 viewer: no smearing, no hole, no tripod/camera artifact
- [ ] No watermark, logo, frame, or text overlay baked in
- [ ] Exposure and WB match the project's other scenes
- [ ] Filename is the final, forever slug (§3)
- [ ] No people/objects cut in half at the ±180 seam

### Common mistakes to avoid

1. **Auto-exposure left on** → every scene has different brightness; transitions flash.
2. **Screen-space effects baked in**: vignette, chromatic aberration, motion blur, depth-of-field, and lens flares are per-frame camera effects — in an equirect they create artifacts, radial darkening, and seam mismatches. Bloom: only if subtle and verified seamless.
3. **Tilted camera** → curved horizon across the whole panorama; unfixable without re-render.
4. **Non-2:1 export** (Lumion UI set to a screen format) → rejected by the validator.
5. **JPEG at default quality** → banding in skies the quality gate will bounce back to you.
6. **Renaming files after delivery** → the slug is a permanent URL; renames break published links.
7. **Wide-gamut color profile** from a post tool → rejected; keep everything sRGB end-to-end.
8. **Post-processing in an external editor that re-saves** with an embedded profile or altered dimensions.

## 2. Project Folder Structure

Create this hierarchy under `content-src/` (masters never deploy; the pipeline reads from here):

```
content-src/
├── <project-slug>/                  ← one folder per project, e.g. villa-horizon/
│   ├── project.authoring.json       ← hand-edited manifest (hierarchy, metadata,
│   │                                   hotspots, views) — engineering supplies the
│   │                                   template at M0.6; you fill in the content
│   ├── panos/                       ← REQUIRED: the panorama masters
│   │   ├── entry-hall.png
│   │   ├── living-room.png
│   │   └── …
│   ├── floorplans/                  ← optional (v1.x, F13): one image per floor
│   │   └── ground-floor.png
│   ├── audio/                       ← optional (v1.x, F16): WAV masters
│   │   └── ambience.wav
│   ├── info-images/                 ← optional: images for information hotspots
│   │   └── facade-detail.png
│   └── overrides/                   ← optional: hand-made poster/thumbnail art,
│       └── …                           only if a generated poster isn't good enough
└── shared/                          ← cross-project brand assets
    ├── logo.svg                     ← wordmark/logo (Phase 3)
    ├── portrait.jpg                 ← about-page portrait (Phase 3)
    └── og-fallback.png              ← social-share fallback image (Phase 4)
```

Only `panos/` + `project.authoring.json` are required per project. Everything else is optional or arrives at later milestones (§8). Never create or edit anything inside `apps/portfolio/public/projects/` by hand — that's pipeline output.

## 3. Scene Naming Rules

**Slugs are forever.** These identifiers become public URLs (`/p/villa-horizon/pano/entry-hall`) and hotspot link targets. Renaming one after publication breaks every shared link to it. Choose names as if carving them in stone.

| Identifier | Becomes | Renameable later? |
|---|---|---|
| Project slug (folder name) | URL path + package folder | ❌ Never after publish |
| Panorama slug (master filename) | URL path + hotspot targets + tile paths | ❌ Never after publish |
| Building / floor / room ids | Manifest references, Space Index anchors | ⚠️ Technically possible pre-publish only; treat as permanent |
| Display names & titles (`"Entry Hall"`) | Visible text only | ✅ Freely, anytime |

Rules for all slugs:

- kebab-case, `a–z`, `0–9`, single hyphens: `entry-hall`, `master-bedroom`, `roof-terrace`
- Descriptive of the *place*, not the render: `living-room`, never `final-v3-test` or `pano-004`
- No dates, versions, or workflow words (`-new`, `-fix`, `-hires`) — re-renders keep the same slug (the pipeline detects the changed content automatically)
- Disambiguate by place, not number, where possible: `living-room-fireplace` / `living-room-terrace-door` beats `living-room-1` / `living-room-2`
- Unique per project across ALL panoramas (the flat namespace spans buildings and floors)

Good: `entry-hall`, `living-room`, `master-bedroom`, `roof-terrace`, `garage-court`, `stair-landing-2f` · Bad: `IMG_0231`, `LivingRoom`, `living_room`, `final-final2`, `pano16k`

## 4. Metadata Requirements

What to prepare per level. **Bold** = required (validation fails without it); the rest is optional but recommended.

### Project
- **id** (slug), **title**, **description** (1–3 sentences; appears on the project page)
- **entrancePanorama** — the slug where the walkthrough starts
- **coverPanorama** — the slug whose poster fronts the project card
- metadata (all optional, all recommended): `author` (your name/studio) · `software` ("Lumion") · `renderEngine` (e.g. "Lumion Ray Tracing") · `year` · `client` · `license` (e.g. "All rights reserved") · `website` (full URL) · `location` (e.g. "Lake Como, Italy") · `categories` (e.g. `["residential", "interior", "exterior"]` — future filtering)

### Building
- **id** (slug), **name** ("Main House") · description (optional). One building? Still define it — the UI simply hides the level.

### Floor
- **id** (slug), **name** ("Ground Floor") · description · `floorplan` (path into `floorplans/`, v1.x)

### Room
- **id** (slug), **name** ("Living Room") · description (shown in the Space Index)

### Panorama
- **id** (slug = filename), **title** ("Living Room — Fireplace")
- **description** — 1–2 sentences. **This is the accessibility text**: it's what a screen-reader visitor gets instead of the imagery. Describe the space, not the render: "Double-height living room with a stone fireplace and lake views to the west."
- **poster alt text** — one sentence describing the poster still (required; build fails without it)
- **initialView** — which direction the visitor faces on arrival (yaw/pitch/fov; authored with the dev picker tool, or described as "facing the fireplace" and translated later)
- Optional: `tags` (`["daylight", "interior"]`) · `createdAt` / `updatedAt` (ISO dates, e.g. `2026-07-09`) · poster/thumbnail overrides (only if generated ones disappoint)

### Hotspot (per link between panoramas)
- **type** — `navigation` (movement) or `information` (commentary panel); `media`/`external` are reserved for the future
- **yaw/pitch position** — where the marker floats (the dev picker tool gives you these numbers by clicking; plan the *connections* now, capture numbers later)
- **label** — the accessible name: "Go to the terrace", "About the facade materials"
- navigation: **target** (panorama slug) + **arrivalView** (the composed view the visitor faces on arrival — treat it like framing a shot)
- information: **title** + **body** text, optional image from `info-images/`

**Prepare now in a spreadsheet or notes:** per project — the hierarchy (buildings → floors → rooms), per panorama — title + description + poster alt, and the connection list ("entry-hall ↔ living-room", "living-room → terrace") with a note on what the visitor should face on each arrival.

## 5. Hotspot Authoring Guide

Decide camera positions and connections **while designing the Lumion shoot** — good transitions are made at render time, not in the manifest.

- **Camera spacing: 2–6 m between connected panoramas.** Under 2 m the step feels pointless; over ~8 m the transition reads as teleportation. Through doorways: one camera either side, 2–4 m apart.
- **Visual overlap is mandatory.** From camera A you must be able to *see* the spot where camera B stands (and ideally vice versa). The Street View illusion = zooming toward a visible destination. Never link two positions separated by a wall.
- **Chain rooms like a walking tour.** Model the graph on how a visitor would actually walk the building: entry → hall → living room → terrace. Add shortcut links sparingly (a grand space may connect to 3–4 others; a corridor to 2).
- **Both directions.** Every connection gets a hotspot in each panorama (A→B and B→A) so visitors can retrace. Place the return hotspot where the previous camera visibly stands.
- **Entrance point:** choose the `entrancePanorama` like a hero shot — it's the first thing every visitor sees, and its poster likely fronts the project card. Approach shots (exterior facing the entry) work beautifully.
- **Arrival views:** for each link, decide what the visitor faces after stepping through — the room's best angle, not a blank wall. Rule of thumb: arrival view ≈ continuing the direction of travel, then adjusted toward the room's composition.
- **Keep hotspots near the horizon.** Markers sit best between pitch −25° and +10° (on the floor plane ahead). Avoid placing navigation near the zenith/nadir.
- **Cross-floor links** happen at stairs/elevators: render a landing panorama so the link is a believable step, not a floor-to-floor jump.
- **Plan on the floor plan:** mark camera positions and arrows on a plan sketch before rendering — it becomes your hotspot map and later the v1.x minimap.

## 6. Lumion Export Guide

| Setting | Recommendation |
|---|---|
| **Output mode** | 360 Panorama (equirectangular), full sphere |
| **Resolution** | 16384 × 8192 (16K); 8192 × 4096 only when 16K is impractical for a scene |
| **Anti-aliasing** | Maximum available. At 16K, aliasing on thin geometry (railings, mullions) survives all the way to the visitor's zoom. |
| **Render quality** | Highest preset / all ray-tracing features your version offers; final-quality shadows and GI |
| **Exposure** | Manual, locked per project. Auto-exposure/eye adaptation **off**. Verify interior scenes don't clip window highlights to pure white across large areas (unrecoverable banding risk). |
| **White balance** | Fixed per project; never auto |
| **Camera height** | 1.60 m, identical every scene (±5 cm) |
| **Camera rotation** | Perfectly level — no roll, no pitch. Only yaw varies. |
| **Safe FOV** | Not applicable to the 360 export itself (it covers the full sphere) — but compose expecting visitors to look everywhere: default viewing FOV in the engine is ~90°, so the "money shot" should read well in a 90° window. |
| **Lighting consistency** | One sun position/sky per project (or one per deliberate set, e.g. "day" and "dusk" versions — which are separate panoramas with their own slugs, not replacements) |
| **Effects to disable** | Vignette · chromatic aberration · motion blur · depth of field · lens flare · film grain · letterboxing/frames · watermarks. These are flat-camera effects; in 360 they produce radial artifacts and seam breaks. |
| **File output** | PNG, sRGB, 8-bit |

## 9. Artist Workflow (per project, once tooling is complete)

```
1. MODELING            Finish the scene; plan camera positions on a floor-plan
                       sketch (§5): spots, connections, arrows = future hotspots.
        ↓
2. LUMION              Place 360 cameras at the planned spots (1.60 m, level);
                       lock exposure/WB/sun once for the whole project (§6).
        ↓
3. EXPORT              Render each camera per §1/§6. Filename = permanent slug.
                       Run the §1 quality checklist on each file.
        ↓
4. FOLDER              content-src/<project-slug>/panos/…  (§2)
   ORGANIZATION        Fill project.authoring.json: hierarchy, metadata,
                       descriptions, alt text, connections (§4).
        ↓
5. ASSET VALIDATION    pnpm validate:content — fix everything it reports
                       (it reports all problems at once, not one at a time).
        ↓
6. PIPELINE            pnpm package:build — tiles, previews, posters, quality
                       gates (banding/seams checked automatically; a rejected
                       scene names the fix). Re-runs are incremental — only
                       changed renders reprocess.
        ↓
7. REVIEW              Open the local viewer; walk every scene: seams, poles,
                       banding, hotspot placement, arrival views.
        ↓
8. WEBSITE             Copy the built package into apps/portfolio/public/projects/,
                       open a content PR — CI re-validates and deploys on merge.
                       No code changes, ever.
```

(Exact command names may consolidate as M0.6/M0.7 land; this document will track them.)

## 10. Production Checklist (final pre-delivery gate)

Before handing any project to the engine:

**Files**
- [ ] Every master: 2:1, 16K (or 8K minimum), PNG, sRGB 8-bit, level horizon, clean poles, seamless ±180 edge
- [ ] No watermarks, overlays, or baked camera effects (§6 disable list)
- [ ] Filenames are final permanent slugs (§3) — reviewed twice, they cannot change
- [ ] Folder layout matches §2 exactly

**Consistency**
- [ ] One camera height across all scenes (±5 cm)
- [ ] One locked exposure + WB per project (or per deliberate set)
- [ ] One sun/sky per project
- [ ] Style-consistent with other portfolio projects where intended

**Content plan**
- [ ] Hierarchy decided: buildings → floors → rooms, each with slug + display name
- [ ] Entrance panorama and cover panorama chosen
- [ ] Connection map complete: every link bidirectional, 2–6 m spacing, visual overlap confirmed (§5)
- [ ] Arrival-view intent noted per link
- [ ] Per panorama: title, description (accessibility text!), poster alt text written
- [ ] Project metadata filled (§4): author, year, location, categories, …

**Handoff**
- [ ] `pnpm validate:content` passes locally (once tooling exists at your machine — otherwise deliver and engineering runs it)
- [ ] You have 30–60 min reserved for the pixel-inspection checkpoint on the built output
